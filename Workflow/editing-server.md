This is exactly the "aim straight at the zombie and whiff" pattern Travis reported, and the same lunge formula applies to jab too (Travis suspected as much but wasn't sure). Fixed by cutting the lunge distances to 8/8/12 (steps 1/2/3) — front-arc hit rate at realistic melee range jumps to 93-99%. Step 4's spin lunge (120, a full 360 spin) is a different mechanic and was untouched. If this regresses, check whether the lunge constants crept back up, or whether `checkSwordHit()`'s blade-capsule math got moved to run before that tick's lunge instead of after (either would reopen the same gap). |
| New/updated art in `images/` doesn't show up in-game even though the file on disk is correct | There used to be a stale duplicate `public/images/` folder (leftover from an older layout, only 4 old files — `spritesheet.png`/`.json`, `Minis.png`/`.json`) that also matched `/images/*` requests. `network.js` registers `express.static('public', ...)` and `express.static('images', ...)` (mounted at `/images`) — static middleware falls through to the next mount only when a file is missing, so whichever mount is registered first wins for any file present in both. With `public` registered first, real edits to `images/spritesheet.png` (e.g. the 2026-07-11 `loot.png` addition) were silently served from the stale copy instead — this is exactly what caused the item-drop world icon to render as a blank/missing sprite (only the rarity glow behind it showed) even though `images/spritesheet.json` had the right frame data. Fixed 2026-07-11: `/images` is now registered before `public`, and the stale `public/images/` folder was deleted outright so this can't silently recur. **The only real image folder is the top-level `images/` — never create or resurrect a `public/images/`.** |
