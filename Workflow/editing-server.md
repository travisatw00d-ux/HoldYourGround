Fed by two sources: drag-and-drop moves (below) and, as of 2026-07-11, zombie item drops (`pickupItem` event, see below). **Not the same thing as `p.inventory`**, which is the older weapon hotbar array (`equip` event, keys 1-9) — see gotcha below. | — |
| Drag-and-drop item moves (bag<->bag, bag<->equip) | `socket-handlers.js` (`moveItem` event -> `room.handleMoveItem`), `room.js` (`handleMoveItem`, broadcasts `playerInfo` only if the move was accepted), `player.js` (`moveItem`/`getItemAtLocation`/`setItemAtLocation`/`canPlaceItem` — location shape `{kind:'bag',index}` or `{kind:'equip',slot}`). Class/type restrictions come from `ITEMS[id].type` (must equal the target slot name) and `ITEMS[id].class` (must equal `p.playerClass`, enforced only for `weapon`/`armor`/`helmet` — rings/necklaces are universal). If the destination is occupied, `moveItem` **swaps** the two items instead of no-oping (2026-07-12 — was a hard no-op before; see gotcha below) — the swap only goes through if the displaced item would also be valid back in the source slot (always true for bag destinations). Fully-rejected moves (type/class mismatch, invalid swap) are still silent no-ops (no error emitted); client has no optimistic update so nothing needs rolling back. | — |

| Item rarity/attribute rolls (tier, rarity, rolled attribute values) | `item-generator.js` (`rollItemRarity`/`getAttributePoolForItem`/`selectRandomAttributes`/`rollAttributeValue`/`generateItemInstance`/`calculateItemStatBonuses`/`resolveBaseItemId`/`isRolledInstance`/`formatItemAttribute` — all data-driven off `ITEM_TIERS`/`ITEM_RARITIES`/`ITEM_ATTRIBUTES` in `config.js`/`shared/data.js`), `player.js` (`equippedStatTotal()` calls `calculateItemStatBonuses()` per equipped item/instance instead of reading `ITEMS[id].stats` directly, `recalcStats()` folds in `maxHealth`/`maxEnergy`/`fortune` from it, `moveItem`/`addToInventory` are instance-shape-agnostic — they store/move whatever's passed in, string or instance), `test-item-generator.js` (validation suite — run after any balance/rule change) | item-generation-system.md |
| Item drop pickup/hover range | `public/shared/data.js`+`lib/game-data.js` (`ITEM_PICKUP_RANGE`, 200 world units — must edit **both** files, same rule as any other shared constant) — server (`item-drops.js`/`room.js`) is the actual authority (`handlePickupItem` silently rejects a pickup from too far away), client (`lib/input.js`'s `hitTestItemDrop()`) mirrors the exact same number so it never shows a hover tooltip or accepts a click for a drop the server would reject anyway — see the "don't reveal what's inside a bag from a distance" gotcha in editing-client.md | editing-client.md |
| Item drops wiped at the start of each wave | `room.js` (`clearItemDrops()` — no-ops if `this.itemDrops` is already empty, otherwise clears it and broadcasts `itemDropsCleared` room-wide), `phase-manager.js` (`_advancePhase()`'s `'daytime'` case calls it right when `matchPhase` flips to `'nighttime'` — i.e. the moment the *next* wave starts). Drops survive the nighttime they spawned in, the following intermission, and the following daytime — they only get wiped when the wave *after* that begins. This is a deliberate design call (2026-07-11), not a bug — don't "fix" drops that are still sitting around mid-daytime. | editing-client.md |

## Gotchas

| Issue | Root cause |
|---|---|
| Queue not promoting | `_promoteFromQueue()` only called on `wasActive` removal (intentional) |
| Late Play Again drops in | Routes through `handleDirectJoin` (not instant respawn) |
| Test commands ignored | Server needs `$env:TEST_MODE=1` to process `__test` |
| Binary packet wrong size | Offsets in `binary-protocol.js` don't match client parse in `net-events.js` |
| Admin commands without login | Every admin socket handler must check `if (!socket.account?.isAdmin) return;` — client can't enforce this |
| Game stats file locked | `game-stats.jsonl` is append-only. Don't read/write simultaneously from multiple handlers. |
| NaN wipes account XP | Always use `Number.isFinite()` guards before DB writes in `_saveRound` (`room.js`). `NaN <= 0` is `false` so bare `if (x <= 0)` doesn't block it. |
| Game-stats lost on deploy | The `stats-tracker.js` LOG_PATH uses `/data/` on Fly.io (persistent volume). Locally it falls back to `data/`. |
| Defense stat has no effect | `p.defense` is computed and displayed (Char Stats panel) but `zombie-ai.js` still applies flat `ZOMBIE_DAMAGE` with no mitigation — wiring defense into damage calc is unbuilt follow-up work. |
| Equipment doesn't survive disconnect/round end | `db.js` `accounts` table only persists `level`/`exp`/`gold`/`cumulative_exp`. `p.equipment`/`currentItem`/`inventory`/`inventorySlots` live only in the in-memory `players[id]` object — reset to `CLASS_LOADOUTS` defaults (bag always resets empty) on every `addPlayer`. No schema/save/load exists yet. |
| New character spawns with an empty Ring/Necklace slot | Intentional, not a missing default — `CLASS_LOADOUTS.knight.ring`/`.necklace` are `null` (2026-07-12, previously `'basic_ring'`/`'basic_necklace'`, two placeholder items with no art and no meaningful stats). They were removed entirely from `ITEMS` because having them pre-equipped silently blocked the *first* real ring/necklace pickup from being dragged in (the slot looked "empty" visually since neither had icon art, but `moveItem` still saw it as occupied) — see the swap-on-drop gotcha below, which is the actual fix for that class of problem, but the filler items were also just not worth keeping around. |
| Dragging a new ring/necklace onto an already-equipped one used to silently fail | `moveItem()` used to reject any move onto an occupied destination outright. Fixed 2026-07-12: it now **swaps** — the new item takes the equip slot, and the item that was there lands in the exact bag slot the new item came from. The swap only happens if the displaced item would also be valid sitting in that source slot (always true when the source is a bag slot, since bag slots accept anything); if you ever build an equip<->equip drag path, remember the swap check runs in both directions. This generalizes to any slot, not just ring/necklace — swapping two bag items, or replacing an equipped armor/weapon/helmet by dragging a new one on top, all use the same code path. |
| Confusing `p.inventory` vs `p.inventorySlots` | Two unrelated arrays with similar names: `p.inventory` is the small weapon hotbar (keys 1-9, `equip` event, `combat-system.js:handleEquip`), unrelated to the new `p.inventorySlots` 16-slot item bag (`addToInventory`). Don't conflate them — a rename was considered but skipped to avoid touching the working hotbar code. |
| Dragging a weapon out of the weapon slot, then pressing a hotbar number key, re-equips it | `moveItem` only touches `p.currentItem`/`p.equipment`/`p.inventorySlots` — it doesn't know about `p.inventory` (the legacy hotbar array `handleEquip` reads from). If a weapon that's still listed in `p.inventory` gets dragged into the bag, pressing its hotbar number re-sets `p.currentItem` from `p.inventory[slot]`, silently re-equipping it (now duplicated: equipped AND sitting in the bag). Not fixed — today `p.inventory` only ever contains the starter weapon and there's no UI to add to it, so it's a narrow edge case; reconciling the two systems is follow-up work if a real hotbar/multi-weapon UI gets built. |
| Clicking an item drop that's far away does nothing | `handlePickupItem()` rejects silently if the player is farther than `item-drops.js`'s `PICKUP_RANGE` (200 world units) from the drop — same "rejected = silent no-op" convention as `moveItem`. The client doesn't pre-check distance itself, it just hit-tests the click against every drop's on-screen label regardless of range, so a far-off click can look like nothing happened rather than an explicit "too far" message. |
| Item drop never spawns even at 50/50 odds over many kills | `rollDropInstance()` (which internally calls `rollForDrop()`) only fires from the `'zombieKilled'` event in `sword.js`'s hit-detection (pushed once per kill, with the zombie's `x`/`y`) — killing a zombie some other way (god-mode kill-all, `killAllMobs()`) doesn't go through that event path and won't roll drops. Intentional for now (admin/debug kills shouldn't farm loot) but worth knowing if drops seem to "not work" during testing with the kill-all debug command. |
| Every drop of the same base item still looks identical in testing | 50% of the time you'll roll Common (1 attribute) on both a ring and a necklace back-to-back — that's expected given the weight table (Common is 50/99.5 ≈ 50.3% of all rolls), not a sign the rarity roll is broken. Run `node server/test-item-generator.js` if you want to sanity-check the actual distribution rather than eyeballing a handful of drops. |
| New/updated art in `images/` doesn't show up in-game even though the file on disk is correct | There used to be a stale duplicate `public/images/` folder (leftover from an older layout, only 4 old files — `spritesheet.png`/`.json`, `Minis.png`/`.json`) that also matched `/images/*` requests. `network.js` registers `express.static('public', ...)` and `express.static('images', ...)` (mounted at `/images`) — static middleware falls through to the next mount only when a file is missing, so whichever mount is registered first wins for any file present in both. With `public` registered first, real edits to `images/spritesheet.png` (e.g. the 2026-07-11 `loot.png` addition) were silently served from the stale copy instead — this is exactly what caused the item-drop world icon to render as a blank/missing sprite (only the rarity glow behind it showed) even though `images/spritesheet.json` had the right frame data. Fixed 2026-07-11: `/images` is now registered before `public`, and the stale `public/images/` folder was deleted outright so this can't silently recur. **The only real image folder is the top-level `images/` — never create or resurrect a `public/images/`.** |
