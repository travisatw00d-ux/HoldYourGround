## Gotchas

| Issue | Root cause |
|---|---|
| Queue not promoting | `_promoteFromQueue()` only called on `wasActive` removal (intentional) |
| Late Play Again drops in | Routes through `handleDirectJoin` (not instant respawn) |
| Test commands ignored | Server needs `$env:TEST_MODE=1` to process `__test` |
| Binary packet wrong size | Offsets in `binary-protocol.js` don't match client parse in `net-events.js` |
| Admin commands without login | Every admin socket handler must check `if (!socket.account?.isAdmin) return;` — client can't enforce this |
| Game stats file locked | `game-stats.jsonl` is append-only. Don't read/write simultaneously from multiple handlers. |
| NaN wipes account XP | Always use `Number.isFinite()` guards before DB writes in `_saveRound` (`room.js`). `NaN <= 0` is `false` so bare `if (x <= 0)` doesn't block it. |
| Game-stats lost on deploy | The `stats-tracker.js` LOG_PATH uses `/data/` on Fly.io (persistent volume). Locally it falls back to `data/`. |
| Defense stat has no effect | `p.defense` is computed and displayed (Char Stats panel) but `zombie-ai.js` still applies flat `ZOMBIE_DAMAGE` with no mitigation — wiring defense into damage calc is unbuilt follow-up work. |
| Fortune stat has no effect | Still intentional as of the 2026-07-13 currency system — a gold-drop mechanic now exists (see the task-map row above), but Fortune isn't wired into it: the drop chance/amount are flat constants in `currency.js`, unaffected by any player stat, per Travis's own spec ("just a thirty percent chance"). Only the stat itself (data model, `recalcStats`, display) exists; don't wire it into the gold roll unless Travis explicitly asks for a Fortune-scaled bonus. Luck (the separate stat right next to it) IS fully wired, but only into equipment rarity odds, not currency — see item-generation-system.md. |
| A returning player's currency briefly shows 0 (or the wrong amount) right after joining/respawning/match-restart | Check that whichever reset path fired (`join-manager.js`'s `handleDirectJoin`/`_promoteFromQueue`, or `phase-manager.js`'s `startMatch()`) still has its matching `room.io.to(id).emit('accountUpdate', ...)` call sending the CURRENT `p.currencyBronze` right after the lvl/exp reset — the client always locally zeroes its currency display on the same `joinedGame`/`eliminated`/match-start events that reset lvl/exp (see editing-client.md), and since currency itself is NOT reset server-side, that local zero is instantly wrong until an accountUpdate corrects it. If a new reset path gets added later without this same emit, this bug reappears there specifically. |
| Currency vanishes / doesn't match what was picked up after a room runs several matches back to back without a player leaving | `_saveRound()`'s delta-save (`this._persistedCurrency` Map, mirrors `_persistedExp`) only flushes the GAIN since the last save, added onto the DB value (`currency_bronze = currency_bronze + ?`) — if `_persistedCurrency` for a player ever gets out of sync with what's actually been persisted (e.g. a new reset path that clears the map entry the way `_persistedExp.delete()` does for exp), the next `_saveRound()` would either re-add already-saved currency (duplication) or skip a real gain (loss). Unlike exp, nothing should ever call `_persistedCurrency.delete()` — currency doesn't reset, so there's no scenario where the tracked baseline needs to be forgotten. If you're tempted to add one by copy-pasting the `_persistedExp.delete()` pattern, don't. |
| Player starts a fresh match at level 1 with a spendable stat point already available | Was a real bug (fixed 2026-07-12): `phase-manager.js`'s `startMatch()` reset `lvl`/`exp`/`gold`/`kills` to fresh-match defaults but restored `statPoints` from `room._persistedExp.get('sp_' + id)` — a leftover-unspent-points cache that `room.js`'s `_saveRound()` wrote at the END of the previous match (`this._persistedExp.set('sp_' + p.id, p.statPoints || 0)`). So any unspent point from a *previous* match carried straight into the next one, before any leveling happened. `_awardExp()` itself was always correct (only grants a point on an actual `levelGain > 0` transition) — the bug was purely this stale-cache restore at match start, not the awarding logic. Fixed by resetting `p.statPoints = 0` at the same point as the other fresh-match fields, and removing the now-dead `'sp_' + p.id` write in `_saveRound()` entirely (nothing else ever read it). If a "carry stat points across matches" feature is ever wanted, it needs to be built deliberately, not left as an accidental side effect of a mid-match exp-persistence cache. |
| Equipment doesn't survive disconnect/round end | Fixed 2026-07-12 for the 5 equip slots (weapon/armor/ring/necklace/helmet) — see the "Persistent equipment" task-map row above. **Still true for the bag** (`p.inventorySlots`) — that's an intentional design choice (only equip-slot items persist), not a gap to close. |
| A logged-in player's equipped gear resets to the starter loadout even though they had something else equipped last time | Check, in order: (1) did `removePlayer()` actually run for their previous session — a TRUE hard crash (OOM-kill, power loss — not a graceful disconnect/leaveRoom, and not a normal deploy/restart either, see the graceful-shutdown row below) could skip it, since the write happens in `_saveEquipment()` at leave time, not continuously during the match; (2) is `accounts.equipment_json` actually populated for that account (`sqlite3 data/accounts.db "SELECT equipment_json FROM accounts WHERE username='...'"`); (3) check the server log for a `[equipment] save failed for account ...` or `[equipment] load failed for account ...` line (added 2026-07-12 — these catches are intentionally still non-throwing, but no longer silent) — a real DB error now leaves a trace instead of looking identical to "never saved." Guests never persist at all — this only applies to logged-in accounts. |
| Deploy/restart loses everyone's in-progress equipment/exp even though `_saveEquipment`/`_saveRound` exist | Fixed 2026-07-12 — `network.js` registers `SIGTERM`/`SIGINT` handlers (`gracefulShutdown()`) that best-effort save every connected account's equipment + exp across every room before the process exits, since Fly.io (and most PaaS platforms) send `SIGTERM` with a grace period before hard-killing on a routine deploy/restart. This does NOT cover a true hard crash (OOM-kill, power loss) — there's no signal to catch in that case — but that failure mode was already non-destructive before this fix and still is: the account's last successfully-saved state is untouched, just not as fresh as it could be. No periodic autosave was added on top of this (deliberate scope decision, discussed with Travis 2026-07-12) — the deploy/restart case covers the realistic scenario; a true hard crash is treated the same as it always has been for exp/gold. |
| Two browser tabs on the same account both save equipment, one clobbers the other | Should no longer be reachable — `activeSessions` (in `socket-handlers.js`) blocks a second `login` for an account that's already got a live socket. Fixed 2026-07-12: the `register` handler now also populates `activeSessions` (it didn't before — only `login` did), which was the actual gap: a freshly-registered account had no `activeSessions` entry until its FIRST login, so a second tab could register/log into that same brand-new account before then without being blocked. If this ever resurfaces, check whether some other auth path (e.g. a future OAuth/SSO flow) sets `socket.account` without also calling `activeSessions.set(...)`. |
| New character spawns with an empty Ring/Necklace slot | Intentional, not a missing default — `CLASS_LOADOUTS.knight.ring`/`.necklace` are `null` (2026-07-12, previously `'basic_ring'`/`'basic_necklace'`, two placeholder items with no art and no meaningful stats). They were removed entirely from `ITEMS` because having them pre-equipped silently blocked the *first* real ring/necklace pickup from being dragged in (the slot looked "empty" visually since neither had icon art, but `moveItem` still saw it as occupied) — see the swap-on-drop gotcha below, which is the actual fix for that class of problem, but the filler items were also just not worth keeping around. |
| Dragging a new ring/necklace onto an already-equipped one used to silently fail | `moveItem()` used to reject any move onto an occupied destination outright. Fixed 2026-07-12: it now **swaps** — the new item takes the equip slot, and the item that was there lands in the exact bag slot the new item came from. The swap only happens if the displaced item would also be valid sitting in that source slot (always true when the source is a bag slot, since bag slots accept anything); if you ever build an equip<->equip drag path, remember the swap check runs in both directions. This generalizes to any slot, not just ring/necklace — swapping two bag items, or replacing an equipped armor/weapon/helmet by dragging a new one on top, all use the same code path. |
| Confusing `p.inventory` vs `p.inventorySlots` | Two unrelated arrays with similar names: `p.inventory` is the small weapon hotbar (keys 1-9, `equip` event, `combat-system.js:handleEquip`), unrelated to the new `p.inventorySlots` 16-slot item bag (`addToInventory`). Don't conflate them — a rename was considered but skipped to avoid touching the working hotbar code. |
| Dragging a weapon out of the weapon slot, then pressing a hotbar number key, re-equips it | `moveItem` only touches `p.currentItem`/`p.equipment`/`p.inventorySlots` — it doesn't know about `p.inventory` (the legacy hotbar array `handleEquip` reads from). If a weapon that's still listed in `p.inventory` gets dragged into the bag, pressing its hotbar number re-sets `p.currentItem` from `p.inventory[slot]`, silently re-equipping it (now duplicated: equipped AND sitting in the bag). Not fixed — today `p.inventory` only ever contains the starter weapon and there's no UI to add to it, so it's a narrow edge case; reconciling the two systems is follow-up work if a real hotbar/multi-weapon UI gets built. |
| Clicking an item drop that's far away does nothing | `handlePickupItem()` rejects silently if the player is farther than `item-drops.js`'s `PICKUP_RANGE` (200 world units) from the drop — same "rejected = silent no-op" convention as `moveItem`. The client doesn't pre-check distance itself, it just hit-tests the click against every drop's on-screen label regardless of range, so a far-off click can look like nothing happened rather than an explicit "too far" message. |
| Item drop never spawns even at 50/50 odds over many kills | `rollDropInstance()` (which internally calls `rollForDrop()`) only fires from the `'zombieKilled'` event in `sword.js`'s hit-detection (pushed once per kill, with the zombie's `x`/`y`) — killing a zombie some other way (god-mode kill-all, `killAllMobs()`) doesn't go through that event path and won't roll drops. Intentional for now (admin/debug kills shouldn't farm loot) but worth knowing if drops seem to "not work" during testing with the kill-all debug command. |
| Every drop of the same base item still looks identical in testing | 50% of the time you'll roll Common (1 attribute) on both a ring and a necklace back-to-back — that's expected given the weight table (Common is 50/99.5 ≈ 50.3% of all rolls), not a sign the rarity roll is broken. Run `node server/test-item-generator.js` if you want to sanity-check the actual distribution rather than eyeballing a handful of drops. |
| Worried a discarded item could get duplicated (still in the bag AND on the ground) | Can't happen by construction, not just by convention: `handleDropItem()` calls `playerMod.setItemAtLocation(p, loc, null)` (clearing the slot) BEFORE `_spawnItemDrop()` (creating the world drop), both synchronously within the same event handler with no `await` in between — Node processes one socket event to completion before the next, so there's no window where both copies exist. This also means a duplicate/double-fired `dropItem` event for the same location is automatically safe: the second call's `getItemAtLocation()` finds the slot already `null` and returns early before reaching `_spawnItemDrop()` at all. Don't "fix" this by adding a debounce/lock on the client or server — it's already race-free. |
| Swing steps 1-2 didn't deal damage to anything caught late in the swing's arc | Was a real bug (fixed 2026-07-12, in `sword.js`'s `checkSwordHit`): `dmgLimit` (which frame-fractions of the swing are still allowed to register a hit) was `halfFrames + 4` for EVERY swing step except the already-special-cased combo3 double-swing — meaning steps 1 and 2 only dealt damage during roughly the first half of their own motion; the return/recovery half of that single swing was pure animation with no damage window at all, even though the blade visually kept moving. Fixed by making `dmgLimit = isSwing ? totalFrames : halfFrames + 1` — every swing step (not just combo3) now deals damage across its ENTIRE motion, forward through return. Jab is untouched (still forward-half-only, `halfFrames + 1`) — that's correct by design, a jab is a quick poke, not a sweeping swing. |
| Swing combo3 (the double-swing) dealt damage on EVERY tick of its return half instead of just once more | A second, related bug caught by the same fix pass (2026-07-12): the hit-loop's per-zombie dedup check used to explicitly bypass the "already hit" guard for the whole back half of combo3 (`cf >= halfFrames`) — intended to let the return swing hit an already-hit target a second time, but with no re-block after that one extra hit, a target sitting in the blade's path for several consecutive ticks during the return swing took damage on EVERY one of those ticks, not just once more. Fixed by removing that bypass entirely — the `_combo3MidHit` reset (below) already clears `attackHitIds` exactly once, at the swing's true midpoint, which is the correct (and now sole) mechanism for "hittable again on the way back." A target can now take at most 2 hits from combo3: one forward, one backward. |
| Swing combo3's forward/backward hit-reset didn't line up with the swing's actual midpoint | Third part of the same fix: the reset that clears `attackHitIds` for combo3's return swing used to fire at `Math.floor(totalFrames / 3)` and wasn't scoped to `isSwing` at all (so it silently also fired, harmlessly-redundantly, during jab's combo3 — jab has its own separate `_jabHitCleared` mechanism for that). Fixed to fire at `halfFrames` (the same "forward motion ends" threshold everything else in this function uses) and scoped to `isSwing`, so the forward/backward split happens exactly where the swing's own animation actually splits, not an arbitrary third of the way through. |
| Swing combo's spin (combo step 4) occasionally gets cut short and the player ends up back at a basic attack | Was a real bug (fixed 2026-07-12): `p.attackStyle`/`p.currentItem` are read LIVE off the player object every tick in `processCombatTick`/`_executeAttack`/`sword.js` (`isUnarmed`, `maxCombo`, `comboKey`, the spin-continuation check `_spinRemaining>0 && p.attackStyle==='swing' && comboStep===4`) — nothing was stopping `toggleAttackStyle`/weapon-hotbar/drag-equip/discard from mutating either mid-combo. The actual failure mode isn't an instant same-tick abort: flipping `attackStyle` mid-spin makes the spin-continuation check fail on the very next tick, which stops `attackFrame` from being pinned back down to 4 every tick (that pin only happens inside the now-skipped spin-continue block) — `attackFrame` then free-runs upward tick by tick until it naturally re-crosses `totalTicks`, at which point the OTHER branch (now evaluating `maxCombo` off the new, wrong `attackStyle`) finally fires and aborts the combo. That delay (anywhere from a couple ticks to most of the remaining spin) is exactly why this looked intermittent/"every once in a while" rather than a reliable, instant repro. Fixed via `isMidCombo(p)` — see the task-map row above; don't reintroduce a live-mutation path for `attackStyle`/`currentItem` without routing it through that same guard. |
| Item drops land at a random offset instead of exactly on the mob that died | `_spawnItemDrop(item, x, y, randomize)` — `handleDropItem()` (player discard, drag-off-inventory / 'x' hotkey) always passes `randomize:true`. History: the random angle+distance offset was added 2026-07-11 to fix discarded items stacking pixel-identical on top of each other when a standing player discarded several in a row (`hitTestItemDrop()`'s tight, closest-only hit test made everything but the top one unclickable) — but the fix was applied UNCONDITIONALLY inside `_spawnItemDrop()`, so it also randomized `emitEvents()`'s `'zombieKilled'` call site, making mob-killed loot land near-but-not-exactly on the mob. Flagged as a regression by Travis 2026-07-13 ("make the drops be positioned back where the mob is that's killed... but make it when a player drops items to have randomness"). Fixed by adding the `randomize` param (default `false`). |
| Two drops from the SAME kill (equipment + gold coin both hit) land stacked on each other | `'zombieKilled'`'s two rolls (equipment instance, gold coin) can both succeed for one kill. A local `killDropCount` counter in that `case` block (2026-07-13, same day as the fix above) makes only the FIRST successful drop pass `randomize:false` (exact mob position) — any drop AFTER that from the same kill passes `randomize:true` via `killDropCount > 0`, same anti-stacking spread `handleDropItem()` uses for repeated discards. Don't revert this to a flat `randomize:false` for both rolls — that reintroduces the exact "can't click what I wanted" complaint that started this whole feature, just from one kill instead of repeated discards. |
| Triple-jab finisher (jab_combo3, comboStep 3) hits inconsistently — same target takes damage way more or way less than once per forward thrust | Was a real bug (fixed 2026-07-13, `sword.js`'s `checkSwordHit`): a redundant `if (isJabFinal) p._jabHitCleared = 0;` used to sit right after the `cfs.length === 0` early return, running on EVERY tick of the attack (not just once at attack start, where `combat-system.js`'s `_executeAttack` already resets it correctly). Since that per-tick reset always ran before the hit-loop, `_jabHitCleared` was back to 0 at the start of nearly every tick from the moment cf first crossed 20 (thrust 2's boundary) onward, so the hit-loop's own `cf >= (jabHitCleared===0 ? 20 : 40)` check kept re-triggering against the `20` branch — `attackHitIds` got wiped almost every tick for the rest of the attack, letting a target that lingered in blade range for several consecutive ticks during a single forward thrust take damage far more than once (confirmed via simulation: 4 hits during thrust 3 alone, should cap at 1). Fixed by deleting that per-tick reset — the attack-start reset in `_executeAttack` is the only one needed, letting `_jabHitCleared` correctly progress 0->1->2 across the whole attack instead of getting stuck resetting to 0 every tick. |
| Swing (and sometimes jab) visibly passes through a zombie standing right in front of you but deals zero damage | Was a real bug (fixed 2026-07-13, in `combat-system.js`'s `_executeAttack`): steps 1-3 lunge the player forward 30 world units (50 on step 3), applied 15/tick in `processCombatTick` AFTER that tick's `checkSwordHit()` call — so on the first tick or two of a swing/jab, the player's own forward rush physically outran the blade's own swing arc, carrying the sword's pivot past a close-range target before the blade ever reached it. Confirmed by simulation (see `outputs/sim/` scratch scripts from that session): with the old 30-unit lunge, a zombie standing dead-ahead at realistic melee range (30-70 world units, i.e. anywhere from just-touching to full swing reach) was only hit 16/37 to 20/37 of the time across the swing's angular sweep; off-center/side targets were largely unaffected. This is exactly the "aim straight at the zombie and whiff" pattern Travis reported, and the same lunge formula applies to jab too (Travis suspected as much but wasn't sure). Fixed by cutting the lunge distances to 8/8/12 (steps 1/2/3) — front-arc hit rate at realistic melee range jumps to 93-99%. Step 4's spin lunge (120, a full 360 spin) is a different mechanic and was untouched. If this regresses, check whether the lunge constants crept back up, or whether `checkSwordHit()`'s blade-capsule math got moved to run before that tick's lunge instead of after (either would reopen the same gap). |
| New/updated art in `images/` doesn't show up in-game even though the file on disk is correct | There used to be a stale duplicate `public/images/` folder (leftover from an older layout, only 4 old files — `spritesheet.png`/`.json`, `Minis.png`/`.json`) that also matched `/images/*` requests. `network.js` registers `express.static('public', ...)` and `express.static('images', ...)` (mounted at `/images`) — static middleware falls through to the next mount only when a file is missing, so whichever mount is registered first wins for any file present in both. With `public` registered first, real edits to `images/spritesheet.png` (e.g. the 2026-07-11 `loot.png` addition) were silently served from the stale copy instead — this is exactly what caused the item-drop world icon to render as a blank/missing sprite (only the rarity glow behind it showed) even though `images/spritesheet.json` had the right frame data. Fixed 2026-07-11: `/images` is now registered before `public`, and the stale `public/images/` folder was deleted outright so this can't silently recur. **The only real image folder is the top-level `images/` — never create or resurrect a `public/images/`.** |
