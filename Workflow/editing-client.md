## Gotchas

| Issue | Root cause |
|---|---|
| Sword/hands invisible | Wrong blade constants in `game-data.js` or missing sprite frames |
| Animation plays wrong arc | `lerpPosePolar` vs `lerpPose` — polar is required for swing arcs |
| Remote sword spins 360° | `shortAngleDelta` not used in blend — will look like long-way spin |
| HUD misaligned | `hud-layout.json` coords wrong or `viewportScale` formula mismatch |
| Client/server item stats disagree | `public/shared/data.js` is a hand-maintained duplicate of `lib/game-data.js` (server can't `require()` an ES module). Edit `ITEMS`/`ITEM_SLOTS`/`CLASS_LOADOUTS`/`BASE_STATS` in **both** files. |
| HUD-sprite popup (charStatsPanel/inventoryPanel) clipped or mispositioned | Their canvas is sized to the trimmed sprite (`dw = spriteSourceSize.w * scale`) and the panel div's left/top already bakes in the trim offset (`sss.x`/`sss.y`). Drawing the sprite into the canvas at `sss.x*s, sss.y*s` again (instead of `0, 0`) double-applies that offset and pushes the art past the canvas edge — this was the bug in both panels as of 2026-07-11. Any new HUD-sprite popup must draw at `(0, 0, dw, dh)`. |
| HUD-sprite popup scale doesn't match hud-position-tool.html | `hud.js`'s always-on HUD elements multiply by `viewportScale` (`vs = 0.471 + 0.529*(viewH/1080)`) to stay legible across window sizes, but `hud-position-tool.html` draws everything raw (`x/y/scale` on a fixed 1024x576 canvas, no `vs`). `showCharStats()`/`showInventory()` in `ui.js` deliberately do **not** multiply by `vs` — they use `layout.scale * wrapperScale` (`wrapperScale = state.viewW / 1024`) so the popup matches the tool exactly at the default 1024x576 wrapper size (`wrapperScale` = 1) instead of rendering ~25% smaller. |
| New HUD-sprite popup's background art shows permanently in a corner, even when its panel div is `.hidden` | `hud.js`'s `drawHUD()` has ONE generic loop that draws every `hudLayout` entry with a matching `hudFrames` image, completely independent of any panel div's visibility — `type:'slot'` entries (InvSlot1-16 etc.) escape this only because they have no matching frame name in HUD.json to look up (`if (!fd) continue`), not because of any explicit exclusion. A real background-art frame (like `inventory.png` or `mctab.png`) DOES have a matching frame, so it WILL get drawn here on every frame unless explicitly added to the `if (el.name === '...') continue;` skip-list a few lines below (currently: `ServerLevel.png`, `settingsgear.png`, `inventory.png`, `Stats.png`, `mctab.png`) — that skip-list is the ONLY thing stopping it. This bit the master chest panel (`mctab.png`) on first ship (2026-07-14) — Travis reported the tab "always up... on the top left of the screen" even before pressing E; root cause was simply forgetting to add `mctab.png` to this list when `inventory.png` was already right there as the template to copy. **Any future HUD-sprite popup whose background art is drawn on-demand into its own dedicated canvas (the `showX()`/`hideX()` pattern) MUST add its frame name to this skip-list**, or it will silently double-render: once correctly (on-demand, panel-gated) and once permanently (unconditionally, every frame, ignoring the panel's hidden state). |
| HUD-sprite popup pinned in top-left corner on fullscreen | `game.js`'s `fullscreenchange` handler resizes `#wrapper` itself to the real screen resolution (not a CSS transform on a fixed 1024x576 box), so any popup positioned with plain 1024x576-design-space pixel coordinates ends up stuck near the corner of a much bigger box. Fixed by the same `wrapperScale = state.viewW / 1024` factor above — it scales *and* repositions the popup proportionally with the wrapper instead of staying pinned. `fullscreenchange` also re-calls `showCharStats()`/`showInventory()` if either panel is currently open so it repositions immediately rather than on next open. `showCharStats()` now also sets `#charStatsContent`'s font-size/line-height/padding inline scaled by `ws = state.viewW / 1024` for the same reason — any new HUD-sprite popup with text content should follow that pattern rather than fixed rem values. |
| Popup content overflows its HUD-sprite panel | `Stats.png` renders at ~224x421px (`inventory.png` similar). `#charStatsContent` needs `position:absolute` with explicit `top/left/right/bottom` (not `padding`) sized to the panel, plus `overflow-y:auto` — the panel (`.char-stats-panel`/`.inventory-panel`) also needs `overflow:hidden` as a backstop. Note: percentage `padding-top` resolves against the container's *width* even on a vertical axis; use `top: X%` instead when you need an offset relative to height (e.g. clearing banner art baked into the top of the sprite). |
| Adding a new interactive slot/hotspot (no sprite of its own) to a HUD panel | `hud-position-tool.html`'s loader only makes a frameless entry draggable/resizable in the tool if it recognizes the element — either hardcode the name (old pattern, e.g. `StatsHotkey`) or, preferred, give it `"type": "slot"` in `hud-layout.json` (new pattern, used for `InvSlot1..16`/`EquipWeapon` etc.). Unlike named-sprite elements, slots have no atlas frame to derive size from, so `w`/`h` **must** be saved in the layout JSON — the tool's `renderOutput()` only includes `w`/`h`/`type` for items where `type === 'slot'`. At runtime, position with `def.x * wrapperScale - panelLeft` (i.e. relative to the panel div, since slots share the same global 1024x576 coordinate space as `inventory.png`/`Stats.png` themselves — see `renderInventorySlots()` in `ui.js`). |
| Sprite looks squished/distorted in `position-tool.html` but fine in-game | `getSpriteFromSheet(sheet, drawW, drawH, frame)` letterboxes its output to match `frame`'s aspect ratio (`frame.w`/`frame.h`, the *trimmed* dimensions) against whatever `drawW`/`drawH` box you pass it — if that box's aspect ratio doesn't match `frame.w`/`frame.h`, the content gets squashed to fit. `render-entity.js` always computes `drawW`/`drawH` as `frame.w * scale`/`frame.h * scale`, so the aspect ratio is identical by construction and there's never any distortion. `position-tool.html`'s Knight Items mode used to compute `drawW`/`drawH` from `frame.sourceSize.w`/`.h` instead (the *untrimmed* canvas size, a totally different aspect ratio for non-square trims like the sword's 53x174) — same bug for scale (sourceSize is usually much bigger than the trimmed frame, so the drag-size and rendered-size didn't match the game either). Fixed 2026-07-11 in `drawKnightMode()` and the two hit-detection blocks in the mousedown/mousemove handlers. **Any position-tool.html mode must use `frame.frame.w`/`.h`, never `frame.sourceSize.w`/`.h`, when computing the destination box passed to `getSpriteFromSheet()`.** |

| Adding icon art for a new item | Add an entry to `ITEM_ICONS` in `lib/game-data.js` (client-only — do **not** add it to `public/shared/data.js`, the server never needs it). If the item's art lives on a sheet not yet loaded, add it to the `Promise.all()` in `lib/assets.js` and a matching `stateKey + 'Sheet'`/`'Frames'` pair in `lib/state.js` — `drawItemIcon()` resolves the sheet dynamically via `state[icon.sheet + 'Sheet']`, so the `sheet` key in `ITEM_ICONS` must exactly match the state property prefix (e.g. `sheet: 'knightWeapons'` -> `state.knightWeaponsSheet`). Items with no `ITEM_ICONS` entry (or whose sheet hasn't loaded yet) fall back to a text label automatically — no error. |
| Adding a new class-restricted item (weapon/armor/helmet) | Set both `type` (must equal the target ITEM_SLOTS name — `'weapon'`/`'armor'`/`'helmet'`) and `class` (must equal a `playerClass` value, e.g. `'knight'`) on the `ITEMS` entry in **both** `lib/game-data.js` and `public/shared/data.js` — the server enforces the class check in `player.js:canPlaceItem`, so a missing/mismatched `class` field silently blocks drag-and-drop equip even though the item still shows fine in the bag. Rings/necklaces should NOT get a `class` field — they're universal accessories, any class can equip them (only `type` is checked for those two slots). |
| Drag-and-drop feels unresponsive / item doesn't move | There's no optimistic client update by design — the drag only sends a `moveItem` socket event on drop; the actual move happens server-side and comes back through the next `playerInfo` broadcast, which `net-events.js` uses to re-run `showInventory()` if the panel's open. If a drop looks like it "did nothing," the server rejected it — wrong item type for that slot, or wrong class. **Dropping onto an already-occupied slot is no longer a no-op** (was, until 2026-07-12) — it swaps the two items, so dragging a new ring/necklace onto an equipped one replaces it and puts the old one in the bag slot the new one came from. See the matching gotcha in editing-server.md for the swap rules. |
| Equipment/inventory changes revert themselves a moment after the drop, or reappear as duplicates | The binary `state` socket event (position/health/etc., ~18x/sec via `BROADCAST_MS`) completely replaces `state.players[id]` every time it arrives — `net-events.js`'s `state` handler builds a brand-new `p` object from scratch per player. Any field not explicitly copied from `state.playerMeta[id]` (the last `playerInfo` payload) into that fresh object is silently dropped on the very next tick, even if `playerInfo` had just set it correctly. This bit both `equipment`/`inventorySlots` (weren't copied at all — a drag result would render once from `playerInfo`, then vanish within ~55ms on the next snapshot) and `currentItem` (was copied with `meta.currentItem \|\| 'wooden_sword'`, which stomps a legitimate `null` — unequipped weapon slot — back to the starter sword, creating an apparent duplicate). Fixed 2026-07-11 by copying `equipment`/`inventorySlots`/`defense`/`turnSpeed`/`playerClass` through every rebuild and switching the `currentItem` fallback to `!== undefined` instead of `\|\|` — `fortune` was added to the same rebuild preventively the same day, and `luck` again 2026-07-12 when that stat was introduced. **Any new per-player field that only comes from `playerInfo` (not the binary protocol) must be added to this same rebuild** or it will silently disappear ~18 times a second. |

| Applying a pasted "Copy Layout JSON" from `hud-position-tool.html` | The tool's export omits falsy/zero fields (e.g. `textX: 0`), so pasting the **full** array back over `hud-layout.json` can silently drop fields on elements you never meant to touch (happened to `PhaseName`/`PhaseTimer`'s `textX`/`textY`/`textSize` on 2026-07-11). When Travis pastes a full layout export, only apply the diff for the specific element(s) he asked to change — diff against the current file and leave every other entry's values exactly as they were, even if the paste appears to alter them. |
| Pressing 'x' drops the wrong item, or does nothing when it looks like something's hovered | `dropHoveredItem()` only acts on `hoveredSlotLoc`, which is set by a slot's `mouseenter` and cleared by that same slot's `mouseleave` (or a full reset on every `renderInventorySlots()` rebuild / `hideInventory()` — see the task-map row above). If the mouse hasn't triggered a fresh `mouseenter` since the last rebuild (e.g. a `playerInfo` broadcast re-rendered the slots layer while the cursor sat still on top of one), `hoveredSlotLoc` is intentionally `null` until the next `mousemove` crosses that element again — this is a deliberate safety reset, not a bug, and means 'x' can occasionally require a tiny mouse nudge right after an inventory change before it "sees" the hover again. |
| 'E' picks up a different item than the one shown in the tooltip | Shouldn't happen post-2026-07-13 fix — both the tooltip and 'E' now read from the exact same `hoveredDropId` variable in `input.js`, set by the same `hitTestItemDrop()` call in `mousemove`, so "what's displayed" and "what E picks up" can't drift apart by construction. If this resurfaces, look for a second/parallel pickup-target calculation being reintroduced (e.g. a "nearest to player" fallback) instead of routing through `hoveredDropId` — that's exactly the bug this fix removed. |
| Inventory item tooltip gets cut off at the bottom/right of the panel | `#itemTooltip` used to live nested inside `#inventoryPanel`, which has `overflow: hidden` (needed to clip the inventory canvas/slots to the panel art) — that also clipped the tooltip whenever it extended past the panel's edges. Fixed 2026-07-11 by moving `#itemTooltip` out to be a sibling of `#dropTooltip` directly under `#wrapper` (same fix `#dropTooltip` already had from the start) and rewriting `positionItemTooltip()` to position off `$.wrapper.getBoundingClientRect()` instead of `$.inventoryPanel`'s. It's allowed to run off the bottom on purpose (no vertical clamp) — the only clamp is horizontal: if the tooltip's width would push it past the wrapper's right edge, it flips to render to the left of the cursor instead. `positionDropTooltip()` (world drops) does **not** have this right-edge flip yet — only the inventory-slot tooltip was reported as an issue, so that one wasn't touched. |
| Drag ghost (dragged item icon) invisible/hard to see while dragging in the inventory | Two compounding causes. (1) `.drag-ghost` in `style.css` had no background/border — for the many items with no `ITEM_ICONS` art yet (`createDragGhost()` in `ui.js` falls back to a bare `.slot-item-label` text node), the "ghost" was just plain text with zero visual container, easy to lose against the game background. (2) `positionGhost()` centered the ghost exactly on the raw cursor coordinates, so the OS pointer icon itself sat right on top of it. Fixed 2026-07-13: `.drag-ghost` now gets a solid dark background/border/border-radius/box-shadow (matching `.item-tooltip`'s box treatment) and `positionGhost()` offsets by new `DRAG_GHOST_OFFSET_X`/`DRAG_GHOST_OFFSET_Y` (14px each) instead of centering — the ghost now trails visibly beside the cursor instead of hiding under it. `onDragEnd()`'s drop-target detection still uses the raw, un-offset `evt.clientX/clientY` via `elementFromPoint()`, so the visual offset doesn't change where a drop actually registers. |
| Gold coins render as a blank/missing icon, or their tooltip is empty | Almost always means `state.itemSheet`/`state.itemFrames` haven't loaded yet, or `ItemSheet.png`/`.json` (containing the `goldcoin.png` frame) never got added to `assets.js`'s `Promise.all()` — check that first, same class of bug as the `public/images/` stale-mount gotcha in editing-server.md. Don't confuse `ItemSheet` (gold coin, misc world items — its own new sheet, 2026-07-13) with `spritesheet.png` (the generic `loot.png` bag icon) — they're two different sheets on purpose; gold intentionally does NOT go through `ITEM_ICONS`/`drawItemIcon()` since it isn't a bag item and has no `ITEMS` entry. |
| Gold-coin drop's hover/click area feels off (too tight, or overlapping a nearby equipment drop) | `hitTestItemDrop()` sizes its rectangle per-drop now (`GOLD_COIN_ICON_H` for gold, `ITEM_DROP_ICON_H` for everything else) — if a fix ever collapses that back down to a single shared `dh`/`dw` computed once outside the loop (the pre-2026-07-13 shape), gold's hit-test would silently start using the much-larger loot-bag size instead of its own small one, or vice versa. Keep the per-drop branch. |
| Currency appears to reset to 0 right after "Play Again"/respawning/joining, even though the server never actually zeroed it | Was a real bug (fixed 2026-07-13) — `net-events.js`'s `'eliminated'`/`'joinedGame'` handlers and `assets.js`'s `enterGame()` all used to locally zero `state.currencyBronze` the same way they zero `state.level`/`state.exp` (copy-pasted from that pattern). That's correct for level/exp — the server genuinely resets those at match restart — but currency is NEVER reset server-side (see editing-server.md's Currency task-map row), so the local zero was always wrong, and in some server paths (`join-manager.js`'s `handleDirectJoin`/`_promoteFromQueue`) it actively raced ahead of and clobbered a correct `accountUpdate` that had just arrived. Fixed by removing the local `currencyBronze = 0` from all three sites — its value is already set correctly by `onAuth()` at login (before any room join happens) and kept current via `accountUpdate`, so there was never a real need to zero it locally in the first place. If a new client-side "reset to defaults on rejoin/respawn" block gets added later, don't copy the level/exp zeroing pattern onto `currencyBronze` — it doesn't apply. |
| Currency text (bronze/silver/gold) looks blurry | Don't add a blurred `text-shadow` (a `0 0 Npx` blur radius) to `.currency-gold`/`.currency-silver`/`.currency-bronze` in `style.css` — that was tried 2026-07-13 as a "glow" and Travis flagged it as making the text unreadable/blurry. The current "shiny" look is a metallic `linear-gradient` painted onto the text via `background-clip: text` + `-webkit-text-fill-color: transparent`, with only a crisp 1px (non-blurred) shadow for legibility. If it ever looks flat/non-shiny in a browser that doesn't support `background-clip: text` on an inline `<span>`, the plain fallback `color` on each class still renders correctly, just without the gradient — don't "fix" that by adding blur back. |
